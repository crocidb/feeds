My experience of the tool in which I write blog posts – Typora – is functionally the same as it was when I started using the tool several years ago. Typora starts as it always does since I set it up: as a blank page, ready for me to start writing.

There have been many updates in that time, but none that have impacted my experience using the application. Typora has many advanced features. Today, I learned the tool has support for showing charts written in the Mermaid markup language. That is to say: features are added, the tool gets better. No update has impacted my core experience or workflow.

Em posted a [list of things she wants and does not want in the technology she uses](https://indieweb.social/@Em0nM4stodon@infosec.exchange/115237473455622491). On the list of “Things I don’t want in my tech”, Em notes “An update that makes it all worse”. This got me thinking about my philosophy for designing the Artemis user interface, from which I have likely drawn some subconscious inspiration from Typora. My general philosophy is the main reading experience should remain the same. The tool should get better, but the main reader should not fundamentally change.

Artemis is regularly updated, but most updates do not impact the core reading experience. I try my best to keep the user experience the same. In fact, the reading experience is almost identical to the way I designed it a year ago, save for the feature that lets users subscribed to ActivityPub posts to see threaded post replies by the same author.

I want Artemis to get better over time – to be faster, to be more reliable, to have more features that make the application pleasant to use (i.e. being able to filter content, or to subscribe to content in different formats). With that said, I have no desire to redesign the user experience. The links of posts written by authors you follow, listed in reverse chronological order, works. I feel much better using this experience than I have in any other reader.

When I add features that do impact the reading interface – for example, when I added a `[hide]` button that lets you hide a post – I make such features opt in so you can add them if you want. The one aberration from this philosophy recently has been the addition of a “search” link in the top navigation; this feature is useful, but it now pushes the navigation bar to two lines on my phone so I see fewer posts on my screen. I have thought about what it would look like for users to be able to customise their navigation bar, although I don’ have a complete idea of what this should look like yet.

I can’t say the Artemis interface will never change, but I can say that I want to be as thoughtful as possible in making changes. If I ever was to completely redesign the UI, I would want to offer everyone the ability to choose between the new and the old. In addition, because Artemis has an API, users can always make their own reader interfaces, whether based on the current styles or your own preferences! (I don’t talk about this enough!)

Like Em, I don’t enjoy when interfaces I am used to change drastically. There have been several occasions in the last two years where software has changed and it has been, at minimum, a bumpy ride – especially those moments when the feature you used daily for years is now elsewhere.
