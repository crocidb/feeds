Last week someone who uses Artemis suggested that it would be nice to be able to use the software with a custom subdomain on their website. This feature suggestion got me thinking about the “[own your links](https://indieweb.org/own_your_links)” concept in the IndieWeb. While the current version of the “own your links” page focuses more on redirects, running software on a custom domain feels relevant to the idea.

If a user could access Artemis with a custom domain, they could move to a different reader that supports custom domains (i.e. self-hosted options) at any point and still keep using the same domain to access their reader. For example, you could access Artemis via `reader.example.com`, where `example.com` is your website.

I have been working on this feature over the last week and now have a version ready for use (in beta).

When Artemis runs on a custom domain, all of the public pages (i.e. the project landing page, the blog) are hidden and redirected to the login page. All custom domains have a restrictive `robots.txt` set up. The login page states that only users authorised to sign in via the custom domain can log in. Password and IndieAuth-based authentication are supported.

When a user logs in to Artemis on a custom domain, their account will be the same as when it is accessed through the primary `artemis.jamesg.blog` domain. This is because, behind the scenes, custom domains use the same running version of Artemis; custom domains don’t run their own separate instance of Artemis.

Starting today, Artemis supporters (anyone who has donated to Artemis) can request to access the service through a custom domain. To get started, email [artemis@jamesg.blog](mailto:artemis@jamesg@.blog).
